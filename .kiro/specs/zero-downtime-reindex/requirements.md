# Requirements Document

## Project Description (Input)
ベクトルDB再インデックスの無停止更新

## Introduction

本機能は、Qdrant 上のアートワーク埋め込みコレクションを**検索サービスを停止させずに再構築・切替**できるようにする。現状は `artworks_v1` という単一コレクションをハードコードで参照しており、タグ体系刷新や埋め込みモデル更新を伴う全件再インデックス時には検索停止か、整合性が壊れた状態でのサービス継続のいずれかを迫られる。

本 spec では Qdrant のコレクションエイリアス機能を用いた Blue/Green 運用を導入し、旧コレクションに対する検索を継続しながら新コレクションへバッチ投入、完走・検証後にエイリアスをアトミックに切り替えるワークフローを整備する。対象はインジェスションサービス、検索サービス、共有 Qdrant リポジトリ、および設定・運用手順。

現時点ではシステムはローカル検証段階であり、本機能導入時の初回カットオーバー（既存 `artworks_v1` 単独運用 → エイリアス運用）では停止を許容する。後方互換や自動移行は要件外とし、無停止はあくまで**導入後の再インデックス運用**に対して求める。

## Requirements

### Requirement 1: エイリアス経由のコレクション参照
**Objective:** 運用者として、物理コレクション名の変更が検索サービスの再デプロイを伴わずに反映できるように、検索サービスが Qdrant エイリアス経由でコレクションを参照する仕組みが欲しい。これによりコレクション切替の自由度を確保する。

#### Acceptance Criteria

1. The 検索サービス shall 読み取り操作（検索・件数取得）をすべて設定で指定されたエイリアス名に対して発行する
2. When 検索サービスが起動する, the システム shall 指定エイリアスが Qdrant 上に存在することを検証し、存在しない場合は明示的なエラーログを出力して起動を失敗させる
3. The システム shall エイリアスと物理コレクションの対応関係を検索リクエスト処理中にキャッシュせず、Qdrant クライアントの同一エンドポイントを通して解決する
4. Where 既存のデフォルト設定値として物理コレクション名 `artworks_v1` が利用されている箇所, the システム shall エイリアス名（デフォルト `artworks_current`）へ置き換え、物理コレクション名の直接参照を検索経路から排除する

### Requirement 2: Blue/Green 物理コレクションの命名と構築
**Objective:** 運用者として、再インデックス先の新コレクションを既存コレクションと衝突せずに構築できるように、バージョン識別可能な命名規則と生成コマンドが欲しい。これにより複数世代のコレクションを並行保持できる。

#### Acceptance Criteria

1. The インジェスションサービス shall 物理コレクション名を `artworks_<version>` 形式（version はユーザー指定、英数字・ハイフン・アンダースコアのみ）で受け付け、指定された名前で新規コレクションを生成する
2. If 指定された物理コレクション名が既に存在する, then the システム shall デフォルトでは処理を中止し、`--force-recreate` 相当のフラグが指定された場合のみコレクションを削除して再作成する
3. While 新コレクションへのバッチ投入が進行している, the システム shall 旧コレクション（現在エイリアスが指している物理コレクション）への書き込み・削除・スキーマ変更を一切行わない
4. The システム shall 新コレクション構築時に、旧コレクションと同一のベクトル次元・距離関数・payload インデックス定義を適用する

### Requirement 3: 原子的なエイリアス切替
**Objective:** 運用者として、切替の瞬間に検索結果が壊れたり一部のリクエストだけ旧コレクションを見たりしないように、エイリアスのアトミックな切替手段が欲しい。これにより整合性を保証したカットオーバーを行える。

#### Acceptance Criteria

1. The システム shall Qdrant の `update_collection_aliases` を単一トランザクションとして発行し、旧エイリアス削除と新エイリアス作成を 1 リクエストで実行する
2. When エイリアス切替が成功する, the システム shall 切替前後の物理コレクション名、切替実行時刻、実行者識別子をログに記録する
3. If エイリアス切替 API 呼び出しが失敗する, then the システム shall エイリアスの状態を変更しないまま終了し、非ゼロ終了コードと原因を返す
4. The システム shall 切替コマンドに `--dry-run` を備え、実際のエイリアス操作を行わず対象の旧/新コレクション名と現在のエイリアス状態を表示する

### Requirement 4: 切替前検証ゲート
**Objective:** 運用者として、不完全な新コレクションへ誤って切り替えないように、切替直前に自動検証が走る仕組みが欲しい。これにより欠損・次元不一致・payload スキーマ差異などの事故を防ぐ。

#### Acceptance Criteria

1. Before エイリアス切替を実行する, the システム shall 新コレクションのポイント件数を取得し、旧コレクションの件数に対して設定された許容比率（デフォルト 0.9 以上）を満たすことを検証する
2. The システム shall 新コレクションに対してサンプル検索（設定で指定された固定クエリ集合）を実行し、全クエリが例外なく結果を返すことを確認する
3. If 検証のいずれかが失敗する, then the システム shall エイリアス切替を中止し、どの検証項目で失敗したかを構造化ログと終了コードで示す
4. Where `--skip-validation` フラグが指定された場合, the システム shall 検証をスキップしつつ、スキップした事実を WARN レベル以上でログに残す

### Requirement 5: ロールバックと旧コレクションの保持
**Objective:** 運用者として、切替直後に問題が発覚したら即座に戻せるように、旧コレクションを一定期間保持し、ワンコマンドで戻せる手段が欲しい。これによりリリース後の障害復旧時間を最小化する。

#### Acceptance Criteria

1. The システム shall エイリアス切替後、旧物理コレクションを自動削除せず、明示的な削除コマンド実行まで保持する
2. The システム shall ロールバックコマンドを提供し、指定された物理コレクションへエイリアスを原子的に戻す
3. If ロールバック対象の物理コレクションが既に削除されている, then the システム shall 切替を中止し、どのコレクションが不在かをエラー出力する
4. The システム shall 古いコレクションの削除コマンドを別途提供し、現在エイリアスが指しているコレクションは誤って削除できないようガードする

### Requirement 6: 追加インジェスションとの両立
**Objective:** 運用者として、再インデックス作業中に新規画像を取りこぼさないように、通常の差分インジェスションと Blue/Green 再インデックスが共存できる設計が欲しい。これにより再インデックス期間中も取り込みパイプラインを止めない。

#### Acceptance Criteria

1. While Blue/Green 再インデックスが進行している, the システム shall 差分インジェスション（新規画像 upsert）を旧コレクション側に対して継続して受け付け、検索にも反映する
2. When エイリアス切替が完了する, the システム shall 切替以降の差分インジェスションを新コレクション側へ向ける
3. The システム shall 切替の前後で、再インデックス対象期間に取り込まれた画像が新コレクションにも反映されるようにキャッチアップ投入手順または再投入スクリプトを提供する
4. If 差分インジェスションの投入先コレクションが現在のエイリアス対象と異なる, then the システム shall 起動時にその旨を WARN ログで通知する

### Requirement 7: 運用可視性（Cloud Logging 互換の構造化ログ）
**Objective:** 運用者として、再インデックスと切替の進行状況をローカル検証でも将来の Google Cloud 本番環境でも同一形式で把握できるように、Cloud Logging が自動解釈できる構造化ログと状態参照 API が欲しい。これによりログ基盤を入れ替えずにローカル→GCP 移行でき、障害時の原因特定と関係者への共有が速くなる。

#### Acceptance Criteria

1. The 検索サービス shall ヘルスチェック応答に、現在エイリアスが解決している物理コレクション名とそのポイント件数を含める
2. The インジェスションサービス shall 再インデックスジョブの進捗（投入件数、失敗件数、経過時間、推定残り時間）を一定間隔（デフォルト 30 秒）で構造化ログ出力する
3. The システム shall すべてのログを標準出力へ 1 行 1 JSON 形式で出力し、Cloud Logging の [structured logging 特殊フィールド](https://cloud.google.com/logging/docs/structured-logging) を以下のとおり採用する: `severity`（`DEBUG`/`INFO`/`NOTICE`/`WARNING`/`ERROR`/`CRITICAL` のいずれか）、`message`（人間可読な要約）、`logging.googleapis.com/labels`（`service`, `env`, `event` 等）、`logging.googleapis.com/trace` と `logging.googleapis.com/spanId`（受信 HTTP ヘッダ `X-Cloud-Trace-Context` または W3C `traceparent` から抽出可能な場合）
4. The システム shall エイリアス切替・ロールバック・検証失敗・差分インジェスション投入先不一致などの運用イベントに対し、`event` ラベルとして一意のイベントキー（例: `reindex.alias.swap`, `reindex.validation.failed`, `reindex.rollback`）を付与し、ログベースメトリクス／アラート定義の着地点として使えるようにする
5. If 例外が補足されずサービス境界まで伝搬する, then the システム shall スタックトレースを Cloud Error Reporting が自動収集可能な形式（`@type: type.googleapis.com/google.devtools.clouderrorreporting.v1beta1.ReportedErrorEvent` とスタックトレース文字列）で 1 レコードに含めて出力する
6. The システム shall ログ設定をコード内にハードコードせず、環境変数（例: `LOG_FORMAT=json|text`, `LOG_LEVEL`）で切り替え可能にして、ローカル開発時の可読ログと本番 JSON ログを同一バイナリで使い分ける

### Requirement 8: Google Cloud 移行を見据えたランタイム前提
**Objective:** 運用者として、本機能をローカル検証後に Terraform で Google Cloud（想定: Qdrant をマネージドまたは GCE/GKE、検索サービスを Cloud Run、再インデックスを Cloud Run Job または GKE CronJob で運用）へ移設する際に、再設計ではなく設定差し替えだけで移行できるように、クラウド非依存のランタイム契約を最初から満たしたい。これにより IaC 導入時の手戻りを防ぐ。

#### Acceptance Criteria

1. The システム shall すべての運用パラメータ（Qdrant エンドポイント、API キー、エイリアス名、物理コレクション名、検証閾値、ログ設定）を環境変数または マウントされた設定ファイルから注入可能にし、コードにハードコードしない
2. The システム shall Qdrant 認証情報（API キー等）をログ・例外メッセージ・ヘルスチェック応答に決して露出させず、Secret Manager からの環境変数注入を前提にした設計とする
3. The 検索サービス shall 再インデックスや切替の途中状態をローカルファイルシステムに永続化せず、必要な状態はすべて Qdrant 自身（コレクション／エイリアス）に保持する（Cloud Run 等のステートレスランタイムで動作する前提）
4. The インジェスションの再インデックスジョブ shall 冪等に起動・再実行できる設計とし、途中失敗後の再実行でも重複なく処理が完了する（Cloud Run Jobs の再試行・手動再実行を想定）
5. The 検索サービス shall liveness/readiness に相当するヘルスチェックエンドポイント（例: `/healthz`, `/readyz`）を提供し、readiness はエイリアス解決が可能な状態でのみ 200 を返す（Cloud Run/GKE のプローブに直接マップできる粒度）
6. The システム shall 将来 Terraform 化する際に Cloud Logging のログベースメトリクスやアラートの入力となる `event` ラベル一覧（Req 7 で定義）を spec の一部として文書化し、IaC 側で参照できるようにする

### Requirement 9: 再インデックス運用ドキュメント（人間 / LLM 共通参照）
**Objective:** 運用者（人間）および AI エージェント（Claude Code 等）として、再インデックスを安全に実行・検証・ロールバックできるように、決定論的な手順と前提条件を 1 つのドキュメントから辿れるようにしたい。これにより属人化と LLM の手順誤認を同時に防ぐ。

#### Acceptance Criteria

1. The システム shall `docs/runbooks/reindex.md`（仮）として、再インデックス・エイリアス切替・ロールバック・旧コレクション削除の各運用フローを 1 ファイルに集約したドキュメントを提供する
2. The ドキュメント shall 各コマンドについて「前提条件」「実行コマンド」「期待ログ（`event` キー）」「成功判定」「失敗時の次アクション」を同一フォーマットで記載する
3. The ドキュメント shall 人間向けの説明文と、LLM が機械的に辿れる構造（見出し階層・コードブロック・イベントキー表）を両立させる（Markdown の見出しとコードフェンスで明示、曖昧な口語表現を避ける）
4. The システム shall プロジェクトのエントリドキュメント（`CLAUDE.md` または `README.md`）から `docs/runbooks/reindex.md` への相対リンクを掲載し、LLM がプロジェクト起動時に発見できるようにする
5. The ドキュメント shall 再インデックス実行時に LLM / 運用者が確認すべきチェックリスト（例: 切替前検証の通過、旧コレクションの残存確認、ヘルスエンドポイント 200 応答）を明記し、コピー可能なコマンドブロック付きで提供する
6. When design.md または CLI のコマンド体系が変更される, the 開発者 shall 同一 PR で `docs/runbooks/reindex.md` を更新する（運用 Runbook をコードと同じ変更単位で扱う）
